# Раздел 19. Тестирование проекта на pytest

## Зачем нужны тесты

Тесты - это код, который проверяет другой код.

В проекте интернет-магазина из раздела 17 уже есть несколько слоев:

- доменные модели;
- репозитории;
- сервис каталога;
- корзина;
- сервис заказов;
- пользователи, адреса и скидки;
- консольное приложение.

Пока проект маленький, его можно проверять вручную: запустить файл, добавить товар, оформить заказ, посмотреть вывод. Но ручная проверка быстро становится ненадежной. После каждого изменения нужно заново помнить все сценарии: корректный заказ, пустая корзина, недостаток товара, неактивный промокод, ошибка базы данных.

Автоматические тесты решают эту проблему. Они фиксируют ожидаемое поведение программы и позволяют быстро понять, сломалось ли что-то после изменения кода.

Тесты особенно полезны в учебном проекте, потому что показывают не только "код запускается", но и "код делает именно то, что должен".

## Основные требования к тестам

Тест должен быть не просто дополнительным кодом рядом с проектом. Хороший тест сам по себе является маленькой спецификацией поведения программы.

Если тест написан плохо, он может мешать сильнее, чем помогать: падать случайно, зависеть от чужих данных, требовать ручного ввода или проверять не то, что действительно важно.

### Независимость

Тесты должны быть независимыми друг от друга.

Это значит:

- один тест не должен требовать, чтобы перед ним обязательно запустился другой тест;
- тест не должен использовать объекты, которые изменил другой тест;
- порядок запуска тестов не должен влиять на результат;
- каждый тест сам готовит нужные данные.

Плохо:

```python
created_product_id = None


def test_create_product(product_repository):
    global created_product_id
    product = Product(1, 1, "Футболка", 1200, "Белый", "База")
    product_repository.add(product)
    created_product_id = product.product_id


def test_get_created_product(product_repository):
    product = product_repository.get_by_id(created_product_id)

    assert product.name == "Футболка"
```

Второй тест зависит от первого. Если запустить его отдельно, он упадет.

Лучше:

```python
def test_get_product_by_id(product_repository):
    product = Product(1, 1, "Футболка", 1200, "Белый", "База")
    product_repository.add(product)

    found_product = product_repository.get_by_id(1)

    assert found_product.name == "Футболка"
```

Этот тест сам создает данные, которые ему нужны.

### Повторяемость

Тест должен давать один и тот же результат при каждом запуске, если код не изменился.

Тест не должен зависеть от:

- текущей даты, если дата не зафиксирована;
- случайных чисел, если они не контролируются;
- ручных данных в базе;
- состояния рабочей базы;
- интернета;
- порядка файлов в папке, если порядок не задан явно;
- пользовательского ввода с клавиатуры.

Например, если тест проверяет промокод с датой окончания, лучше передать дату явно или подменить текущую дату, а не надеяться на сегодняшний день.

### Изоляция

Тест должен проверять ровно тот уровень, который заявлен.

Если это unit-тест сервиса каталога, ему обычно не нужна настоящая PostgreSQL-база. Можно использовать fake-репозиторий.

Если это интеграционный тест репозитория, тогда настоящая тестовая база нужна, но она должна быть отдельной от рабочей.

Изоляция помогает быстрее понять причину падения:

- упал unit-тест модели - проблема в модели;
- упал unit-тест сервиса - проблема в бизнес-логике;
- упал интеграционный тест репозитория - проблема в SQL, схеме или подключении;
- упал полный сценарий - проблема может быть в связке нескольких слоев.

### Самопроверяемость

Тест должен сам проверять результат через `assert` или через ожидаемое исключение.

Плохо:

```python
def test_cart_total(cart):
    print(cart.total_sum())
```

Такой тест ничего не проверяет. Человек должен глазами смотреть вывод.

Лучше:

```python
def test_cart_total(cart):
    assert cart.total_sum() == 6000
```

Если ожидается ошибка, ее тоже нужно проверять явно:

```python
def test_cart_rejects_negative_quantity(cart, product):
    with pytest.raises(ValueError):
        cart.add_item(product=product, size="M", quantity=-1)
```

### Понятность

Название теста должно объяснять поведение, которое проверяется.

Плохо:

```python
def test_1():
    ...


def test_product():
    ...
```

Лучше:

```python
def test_product_rejects_negative_price():
    ...


def test_cart_adds_same_product_size_as_one_position():
    ...


def test_order_service_does_not_save_order_when_stock_is_not_enough():
    ...
```

Когда тест падает, его название должно уже подсказывать, что именно сломалось.

### Фокус на поведении

Тесты лучше проверяют поведение программы, а не внутренние детали реализации.

Если сервис каталога должен возвращать только активные товары, тесту важно проверить результат:

```python
result = service.get_active_products()

assert inactive_product not in result
assert active_product in result
```

Не стоит без необходимости проверять, какие временные переменные сервис создал внутри метода. Внутреннюю реализацию можно поменять, а поведение должно остаться тем же.

### Маленький размер

Один тест должен проверять одну идею.

Если тест одновременно проверяет создание товара, поиск, корзину, скидку, заказ и вывод меню, при падении будет сложно понять причину.

Лучше иметь несколько небольших тестов:

- товар запрещает отрицательную цену;
- каталог ищет товар без учета регистра;
- корзина считает сумму;
- скидка применяется к подходящей сумме;
- заказ списывает остатки после успешного оформления.

Полный пользовательский сценарий тоже нужен, но он должен быть отдельным интеграционным тестом, а не заменой всех маленьких тестов.

### Скорость

Тесты должны запускаться достаточно быстро, чтобы их хотелось запускать часто.

Обычно:

- unit-тесты должны быть очень быстрыми;
- интеграционные тесты с PostgreSQL могут быть медленнее;
- полный сценарий может быть еще медленнее, но таких тестов должно быть меньше.

Если весь набор тестов запускается слишком долго, разработчики начинают запускать его редко. Тогда тесты перестают выполнять свою главную работу - быстро предупреждать об ошибках.

### Аккуратная работа с внешним состоянием

Внешнее состояние - это все, что живет вне одного теста:

- база данных;
- файлы;
- переменные окружения;
- сетевые запросы;
- текущая дата и время;
- глобальные переменные;
- пользовательский ввод.

С таким состоянием нужно работать осторожно:

- использовать отдельную тестовую базу;
- очищать данные перед тестом или откатывать транзакцию после теста;
- создавать временные файлы через `tmp_path`;
- подменять переменные окружения через `monkeypatch`;
- не использовать `input()` в сервисах и моделях;
- не проверять бизнес-логику через настоящую консоль, если можно проверить сервис напрямую.

### Детерминированность

Детерминированный тест - это тест, который при одинаковом коде и одинаковых входных данных всегда дает одинаковый результат.

Недетерминированные тесты особенно неприятны: сегодня они проходят, завтра падают, потом снова проходят. Такие падения сложно расследовать, и команда перестает доверять тестам.

Частые причины случайных падений:

- тест зависит от текущего времени;
- тест использует случайные числа без фиксированного seed;
- тест зависит от данных, оставшихся после прошлых запусков;
- тест обращается к рабочей базе;
- тест ожидает один порядок элементов, хотя код его не гарантирует.

Если порядок важен, задайте сортировку явно. Если важен набор элементов без порядка, сравнивайте множества или отсортированные списки.

### Проверка важных сценариев

Тесты должны покрывать не только идеальный сценарий.

Для проекта магазина важно проверять:

- happy path - все данные корректны, операция успешна;
- negative path - пользователь или система дают неправильные данные;
- edge cases - пограничные значения;
- regression tests - ошибки, которые уже находили и исправляли;
- integration tests - связку нескольких слоев;
- transaction tests - отсутствие частично сохраненных данных при ошибке.

Особенно внимательно нужно тестировать код, который меняет данные: оформление заказа, списание остатков, применение скидки, сохранение в PostgreSQL.

## pytest

`pytest` - популярный фреймворк для тестирования Python-кода.

Он удобен тем, что:

- тесты пишутся обычными функциями;
- для проверки используется обычный `assert`;
- есть удобная работа с ошибками;
- есть фикстуры для подготовки данных;
- есть параметризация, чтобы проверять много похожих случаев;
- хорошо подходят как unit-тесты, так и интеграционные тесты.

Установка:

```bash
python -m pip install pytest
```

Для интеграционных тестов с PostgreSQL используйте тот же драйвер, что и в разделе 17:

```bash
python -m pip install psycopg2-binary
```

В одном проекте не нужно смешивать разные драйверы PostgreSQL. Если основной код подключается к PostgreSQL через `psycopg2`, тесты тоже должны использовать `psycopg2`.

Запуск всех тестов из корня проекта:

```bash
python -m pytest
```

Краткий вывод:

```bash
python -m pytest -q
```

Запуск одного файла:

```bash
python -m pytest tests/test_domain_models.py
```

Запуск тестов по части названия:

```bash
python -m pytest -k product
```

Остановиться после первой ошибки:

```bash
python -m pytest -x
```

```python
def test_product_accepts_correct_data():

```

## Из чего состоит тест

Хороший тест обычно устроен по схеме AAA:

1. Arrange - подготовка данных.
2. Act - выполнение действия.
3. Assert - проверка результата.

Пример:

```python
def test_product_price_can_be_changed():
    product = Product(
        product_id=1,
        category_id=1,
        name="Футболка",
        price=1200,
        color="Белый",
        description="Базовая футболка",
    )

    product.change_price(1500)

    assert product.price == 1500
```

В этом тесте:

- Arrange: создаем товар;
- Act: меняем цену;
- Assert: проверяем новую цену.

Иногда используют похожую схему Given/When/Then:

- Given - дано некоторое состояние;
- When - когда происходит действие;
- Then - тогда ожидаем результат.

Смысл тот же: тест должен быть читаемым как маленький сценарий.

## Один тест - одна идея

Тест не должен проверять сразу половину приложения.

Плохо:

```python
def test_everything():
    # создать товар
    # добавить в корзину
    # применить скидку
    # оформить заказ
    # проверить репозиторий
    # проверить вывод меню
```

Такой тест сложно понять. Если он упал, непонятно, какая часть сломалась.

Лучше:

```python
def test_product_rejects_negative_price():
    ...


def test_cart_adds_product_with_size():
    ...


def test_discount_service_applies_active_promo_code():
    ...


def test_order_service_reduces_stock_after_successful_order():
    ...
```

Каждый тест проверяет одну мысль.

## Happy path

Happy path - это сценарий, в котором все идет правильно.

Для магазина одежды happy path может быть таким:

1. Товар существует.
2. Товар активен.
3. Нужный размер есть на складе.
4. Покупатель существует.
5. Корзина не пустая.
6. Промокод корректный.
7. Заказ успешно создается.
8. Остатки товара уменьшаются.

Пример unit-теста happy path для модели остатка:

```python
def test_product_stock_removes_available_quantity():
    stock = ProductStock(
        stock_id=1,
        product_id=1,
        size="M",
        quantity=5,
    )

    stock.remove_quantity(2)

    assert stock.quantity == 3
```

Пример happy path для корзины:

```python
def test_cart_total_sum_uses_quantity_and_price():
    cart = Cart(customer_id=1)
    product = Product(
        product_id=1,
        category_id=1,
        name="Худи",
        price=3000,
        color="Серый",
        description="Теплое худи",
    )

    cart.add_item(product=product, size="M", quantity=2)

    assert cart.total_sum() == 6000
```

Названия классов и методов могут отличаться в вашей реализации. В тестах нужно использовать имена из вашего кода.

## Negative path

Negative path - это сценарий, в котором пользователь или система делают что-то неправильное.

В таких тестах мы проверяем, что программа не молча ломается, а ведет себя предсказуемо:

- поднимает понятное исключение;
- возвращает `None`;
- возвращает пустой список;
- не изменяет данные;
- откатывает транзакцию.

Пример:

```python
import pytest


def test_product_rejects_negative_price():
    with pytest.raises(ValueError):
        Product(
            product_id=1,
            category_id=1,
            name="Футболка",
            price=-100,
            color="Белый",
            description="Базовая футболка",
        )
```

Если код внутри `with pytest.raises(ValueError)` не вызовет `ValueError`, тест упадет.

Можно проверить текст ошибки:

```python
def test_product_rejects_empty_name():
    with pytest.raises(ValueError, match="Название"):
        Product(
            product_id=1,
            category_id=1,
            name="",
            price=1200,
            color="Белый",
            description="Базовая футболка",
        )
```

Текст ошибки не стоит проверять слишком жестко. Обычно достаточно ключевого слова.

## Edge cases

Edge cases - пограничные случаи.

Они не всегда ошибочные, но часто ломают код.

Примеры для магазина:

- цена равна `0`;
- остаток товара равен `0`;
- покупатель добавляет ровно все доступное количество;
- промокод дает скидку `0%`;
- промокод дает максимально разрешенную скидку;
- поиск идет по нижнему регистру, а товар записан с заглавной буквы;
- корзина очищается после оформления заказа;
- заказ состоит из одной позиции;
- заказ состоит из нескольких позиций одного товара разных размеров.

Пример:

```python
def test_stock_allows_removing_exact_available_quantity():
    stock = ProductStock(
        stock_id=1,
        product_id=1,
        size="L",
        quantity=3,
    )

    stock.remove_quantity(3)

    assert stock.quantity == 0
```

## Regression tests

Regression test - тест на уже найденную ошибку.

Например, в проекте обнаружили баг: поиск товара по названию работает только если пользователь ввел текст в том же регистре, что и в базе.

После исправления нужно добавить тест:

```python
def test_catalog_search_is_case_insensitive():
    result = catalog_service.search_by_name("худи")

    assert len(result) == 1
    assert result[0].name == "Худи Oversize"
```

Теперь если кто-то случайно вернет старую ошибку, тест упадет.

## Unit-тесты

Unit-тест проверяет маленькую часть программы отдельно от остальных.

В проекте магазина unit-тестами удобно покрывать:

- доменные модели;
- методы корзины;
- расчет скидки;
- фильтрацию и сортировку в сервисах;
- бизнес-правила оформления заказа.

Unit-тесты должны быть быстрыми. Обычно они не ходят в настоящую базу данных и не требуют `input()`.

Пример:

```python
def test_promo_code_applies_percent_discount():
    promo = PromoCode(
        code="SALE10",
        discount_percent=10,
        is_active=True,
        min_order_sum=1000,
    )

    service = DiscountService()

    result = service.apply_promo_code(total_sum=2000, promo_code=promo)

    assert result == 1800
```

Если в вашей реализации метод возвращает не число, а объект с деталями скидки, проверяйте его поля.

## Интеграционные тесты

Интеграционный тест проверяет несколько частей вместе.

В проекте магазина интеграционными тестами стоит покрыть:

- репозиторий + PostgreSQL;
- сервис заказов + корзина + репозитории;
- оформление заказа + списание остатков + сохранение заказа;
- применение скидки в заказе.

Интеграционные тесты медленнее unit-тестов, зато они проверяют реальные связи между слоями.

Пример сценария:

1. Подготовить тестовую базу.
2. Добавить категорию, товар, остаток и покупателя.
3. Добавить товар в корзину.
4. Оформить заказ.
5. Проверить, что заказ сохранен.
6. Проверить, что остаток уменьшился.

Важно: интеграционные тесты не должны портить рабочую базу.

Для тестов лучше использовать отдельную базу, например:

```text
clothing_store_test
```

А рабочую:

```text
clothing_store
```

## Тестовая база данных

Для репозиториев и заказов нужна отдельная тестовая PostgreSQL-база.

Принципы:

- тесты не должны использовать рабочую базу;
- перед тестом нужно подготовить чистое состояние;
- после теста данные нужно удалить или откатить;
- тест должен быть повторяемым.

Один из подходов - фикстура соединения с транзакцией:

```python
import os
import pytest
import psycopg2


@pytest.fixture
def connection():
    conn = psycopg2.connect(os.environ["TEST_DATABASE_URL"])
    conn.autocommit = False

    yield conn

    conn.rollback()
    conn.close()
```

Все изменения внутри теста будут отменены после `yield`.

Если ваш код сам вызывает `commit()`, простого `rollback()` может быть недостаточно. Тогда можно:

- очищать таблицы перед каждым тестом через `DELETE`;
- пересоздавать схему перед запуском тестов;
- использовать отдельную базу, которую не жалко очищать;
- изменить репозитории так, чтобы управление транзакцией было на уровне сервиса.

Для учебного проекта допустимо выбрать простой подход: перед каждым интеграционным тестом очищать таблицы тестовой базы.

Пример фикстуры очистки:

```python
@pytest.fixture
def clean_database(connection):
    with connection.cursor() as cursor:
        cursor.execute("DELETE FROM order_items")
        cursor.execute("DELETE FROM orders")
        cursor.execute("DELETE FROM product_stocks")
        cursor.execute("DELETE FROM products")
        cursor.execute("DELETE FROM categories")
        cursor.execute("DELETE FROM customers")
    connection.commit()

    return connection
```

Порядок удаления важен: сначала дочерние таблицы, потом родительские.

## Фикстуры

Фикстура - функция, которая готовит данные или зависимости для теста.

Пример:

```python
import pytest


@pytest.fixture
def product():
    return Product(
        product_id=1,
        category_id=1,
        name="Футболка",
        price=1200,
        color="Белый",
        description="Базовая футболка",
    )


def test_product_can_be_deactivated(product):
    product.deactivate()

    assert product.is_active is False
```

`pytest` сам передаст результат фикстуры в тест по имени аргумента.

Фикстуры помогают убрать повторяющуюся подготовку.

## Область жизни фикстуры

По умолчанию фикстура создается заново для каждого теста.

Это хорошо: один тест не должен случайно менять данные другого теста.

Можно указать `scope`:

```python
@pytest.fixture(scope="module")
def expensive_data():
    ...
```

Основные варианты:

- `function` - для каждого теста, значение по умолчанию;
- `module` - один раз на файл;
- `session` - один раз на весь запуск pytest.

Для объектов, которые тесты меняют, лучше использовать обычный `function` scope.

## Параметризация

Параметризация позволяет одним тестом проверить несколько похожих случаев.

Без параметризации приходится писать несколько почти одинаковых тестов:

```python
def test_product_rejects_empty_name():
    ...


def test_product_rejects_empty_color():
    ...


def test_product_rejects_empty_description():
    ...
```

Это работает, но код быстро разрастается. Если подготовка объекта большая, повторение начинает мешать читать тесты.

`@pytest.mark.parametrize` позволяет передать в один тест разные наборы данных.

Общий вид:

```python
@pytest.mark.parametrize(
    "argument_1,argument_2,expected",
    [
        (value_1, value_2, expected_1),
        (value_3, value_4, expected_2),
    ],
)
def test_something(argument_1, argument_2, expected):
    assert some_function(argument_1, argument_2) == expected
```

Первая строка перечисляет имена аргументов теста. Список ниже содержит наборы значений. `pytest` запустит тест столько раз, сколько наборов данных указано.

Например, модель товара должна запрещать пустые строки в нескольких полях:

```python
import pytest


@pytest.mark.parametrize(
    "name,color,description",
    [
        ("", "Белый", "Базовая футболка"),
        ("Футболка", "", "Базовая футболка"),
        ("Футболка", "Белый", ""),
    ],
)
def test_product_rejects_empty_text_fields(name, color, description):
    with pytest.raises(ValueError):
        Product(
            product_id=1,
            category_id=1,
            name=name,
            price=1200,
            color=color,
            description=description,
        )
```

Так тест остается коротким, но проверяет несколько вариантов.

Параметризация хорошо подходит для проверки допустимых размеров:

```python
@pytest.mark.parametrize("size", ["XS", "S", "M", "L", "XL", "XXL"])
def test_product_stock_accepts_allowed_sizes(size):
    stock = ProductStock(
        stock_id=1,
        product_id=1,
        size=size,
        quantity=3,
    )

    assert stock.size == size
```

И для проверки недопустимых размеров:

```python
@pytest.mark.parametrize("size", ["", "A", "42", "XXXL", "small"])
def test_product_stock_rejects_unknown_sizes(size):
    with pytest.raises(ValueError, match="размер"):
        ProductStock(
            stock_id=1,
            product_id=1,
            size=size,
            quantity=3,
        )
```

Можно параметризовать не только входные данные, но и ожидаемый результат:

```python
@pytest.mark.parametrize(
    "price,quantity,expected_total",
    [
        (1000, 1, 1000),
        (1000, 3, 3000),
        (2500, 2, 5000),
    ],
)
def test_cart_item_total_sum(price, quantity, expected_total):
    product = Product(
        product_id=1,
        category_id=1,
        name="Футболка",
        price=price,
        color="Белый",
        description="Базовая футболка",
    )
    item = CartItem(product=product, size="M", quantity=quantity)

    assert item.total_sum() == expected_total
```

Для скидок параметризация особенно удобна, потому что у промокода много условий:

```python
@pytest.mark.parametrize(
    "total_sum,discount_percent,min_order_sum,expected",
    [
        (2000, 10, 1000, 1800),
        (1000, 10, 1000, 900),
        (900, 10, 1000, 900),
        (5000, 0, 1000, 5000),
    ],
)
def test_discount_service_calculates_total_sum(
    total_sum,
    discount_percent,
    min_order_sum,
    expected,
):
    promo_code = PromoCode(
        code="SALE",
        discount_percent=discount_percent,
        is_active=True,
        min_order_sum=min_order_sum,
    )
    service = DiscountService()

    result = service.apply_promo_code(total_sum, promo_code)

    assert result == expected
```

Если тестов много, удобно добавлять `ids`. Тогда в выводе pytest будет видно не просто номер набора данных, а понятное название случая:

```python
@pytest.mark.parametrize(
    "quantity",
    [0, -1, -10],
    ids=["zero", "minus_one", "minus_ten"],
)
def test_cart_rejects_invalid_quantity(quantity, product):
    cart = Cart(customer_id=1)

    with pytest.raises(ValueError, match="количество"):
        cart.add_item(product=product, size="M", quantity=quantity)
```

Параметризацию не стоит применять механически везде. Если сценарии сильно отличаются по смыслу, лучше написать отдельные тесты с понятными названиями. Она особенно полезна там, где логика одна, а входных значений несколько.

## Проверка исключений

Ошибочные сценарии часто проверяют через `pytest.raises`.

Пример для корзины:

```python
def test_cart_rejects_zero_quantity(product):
    cart = Cart(customer_id=1)

    with pytest.raises(ValueError, match="количество"):
        cart.add_item(product=product, size="M", quantity=0)
```

Пример для заказа:

```python
def test_order_service_rejects_empty_cart(order_service, empty_cart, customer):
    with pytest.raises(ValueError, match="корзина"):
        order_service.create_order(customer, empty_cart)
```

Главная мысль: ошибка - это тоже ожидаемое поведение, если программа явно его описывает.

## Test doubles: fake, stub, mock

Не всегда нужно подключать настоящую базу.

Иногда сервис можно проверить с поддельным репозиторием.

Test double - общий термин для подмены настоящей зависимости в тесте.

Частые варианты:

- fake - простая рабочая замена, например репозиторий на списке;
- stub - объект, который возвращает заранее заданные ответы;
- mock - объект, у которого проверяют вызовы методов.

Пример fake-репозитория:

```python
class FakeProductRepository:
    def __init__(self, products):
        self.products = products

    def get_all(self):
        return self.products

    def get_by_id(self, product_id):
        for product in self.products:
            if product.product_id == product_id:
                return product
        return None
```

Так можно тестировать `CatalogService` без PostgreSQL.

Пример:

```python
def test_catalog_service_returns_only_active_products():
    active = Product(1, 1, "Футболка", 1200, "Белый", "База", True)
    inactive = Product(2, 1, "Архивный товар", 900, "Черный", "Старый", False)
    repository = FakeProductRepository([active, inactive])
    service = CatalogService(repository)

    result = service.get_active_products()

    assert result == [active]
```

## Mock

`mock` полезен, когда нужно проверить не только результат, но и факт вызова зависимости.

В Python есть стандартный модуль `unittest.mock`.

Пример:

```python
from unittest.mock import Mock


def test_order_service_saves_order(order):
    order_repository = Mock()
    stock_repository = Mock()
    service = OrderService(order_repository, stock_repository)

    service.save_order(order)

    order_repository.add.assert_called_once_with(order)
```

Mock не стоит использовать везде. Если можно проще проверить результат через обычный fake-объект, лучше начать с fake.

## monkeypatch

`monkeypatch` - встроенная фикстура pytest для временной подмены значений.

Например, можно подменить переменную окружения:

```python
def test_database_url_is_read_from_environment(monkeypatch):
    monkeypatch.setenv("TEST_DATABASE_URL", "postgresql://user:pass@localhost/db")

    assert os.environ["TEST_DATABASE_URL"].endswith("/db")
```

Можно подменить функцию:

```python
def test_app_uses_test_connection(monkeypatch):
    fake_connection = object()

    monkeypatch.setattr(storage_module, "get_connection", lambda: fake_connection)

    assert storage_module.get_connection() is fake_connection
```

После теста pytest сам вернет старое значение.

## capsys

`capsys` позволяет проверить вывод в консоль.

Это полезно для финального консольного приложения, но лучше не строить всю проверку только на тексте вывода.

Пример:

```python
def test_print_catalog_outputs_product_name(capsys):
    print_catalog([Product(1, 1, "Футболка", 1200, "Белый", "База")])

    captured = capsys.readouterr()

    assert "Футболка" in captured.out
```

Сервисы и модели лучше проверять по возвращаемым значениям, а не по `print()`.

## tmp_path

`tmp_path` дает временную папку для теста.

Это полезно, если в проекте появляются файлы: логи, экспорт заказа, временные настройки.

```python
def test_order_export_creates_file(tmp_path):
    file_path = tmp_path / "order.txt"

    export_order(order, file_path)

    assert file_path.exists()
```

Для текущего проекта с PostgreSQL `tmp_path` может не понадобиться, но полезно знать инструмент.

## Импорт модулей из раздела 17

Папки курса начинаются с цифр, поэтому обычный импорт вида:

```python
from 17_clothing_store_project.01_domain_models.tasks import Product
```

не подходит.

В проекте уже используется `importlib.import_module()`.

В тестах можно делать так же:

```python
from importlib import import_module


domain_models = import_module("17_clothing_store_project.01_domain_models.tasks")

Product = domain_models.Product
ProductStock = domain_models.ProductStock
Customer = domain_models.Customer
```

Если тесты лежат в папке `tests/`, запускайте pytest из корня репозитория:

```bash
python -m pytest
```

Так Python сможет найти папку `17_clothing_store_project`.

## Что именно тестировать в разделе 17

### Доменные модели

Проверить:

- корректное создание категории, товара, остатка, покупателя;
- запрет отрицательной цены;
- запрет пустого названия;
- запрет неизвестного размера;
- доступность остатка;
- списание остатка;
- ошибка при попытке списать больше, чем есть.

Типы тестов:

- happy path;
- negative path;
- edge cases.

### Репозитории

Проверить:

- добавление записи;
- получение по идентификатору;
- получение списка;
- обновление;
- удаление;
- поведение при несуществующем идентификаторе;
- сохранение и чтение остатков по размерам;
- сохранение и чтение заказов.

Для репозиториев лучше использовать интеграционные тесты с тестовой PostgreSQL-базой.

### Сервис каталога

Проверить:

- возвращаются только активные товары;
- поиск по части названия не зависит от регистра;
- фильтр по категории;
- фильтр по цвету;
- фильтр по размеру учитывает остаток;
- фильтр по цене;
- сортировка по цене;
- сортировка по названию;
- пустой результат возвращается как пустой список, а не как ошибка.

Для сервиса каталога можно использовать fake-репозитории.

### Корзина

Проверить:

- товар добавляется в корзину;
- повторное добавление того же товара и размера увеличивает количество;
- разные размеры одного товара хранятся отдельно;
- нельзя добавить нулевое или отрицательное количество;
- нельзя добавить больше, чем есть на складе;
- изменение количества работает;
- удаление позиции работает;
- итоговая сумма считается правильно;
- очистка корзины удаляет позиции.

### Заказы

Проверить:

- заказ создается из непустой корзины;
- позиции заказа фиксируют цену на момент покупки;
- после успешного заказа корзина очищается;
- остатки уменьшаются именно по выбранному размеру;
- пустая корзина приводит к ошибке;
- нехватка товара приводит к ошибке;
- при ошибке заказ не сохраняется частично;
- при ошибке остатки не списываются частично;
- история заказов покупателя возвращает его заказы.

Заказы - один из самых важных участков для тестов, потому что здесь есть транзакции.

### Пользователи и скидки

Проверить:

- адрес доставки создается с корректными данными;
- промокод применяется к подходящей сумме;
- неактивный промокод не применяется;
- неизвестный промокод не применяется;
- промокод не применяется, если сумма меньше минимальной;
- итоговая сумма не становится отрицательной;
- заказ хранит исходную сумму, скидку и итоговую сумму.

### Консольное приложение

Проверить:

- функции интерфейса не создают бизнес-логику внутри себя;
- пользовательский ввод можно подменить в тесте;
- ключевые сообщения выводятся в консоль;
- при неверном вводе программа не падает.

Но консольный слой обычно тестируют меньше, чем модели и сервисы. Главная логика должна быть в сервисах, где ее проще проверить.

## Тесты и транзакции

Оформление заказа должно быть атомарным.

Атомарность означает: операция выполняется целиком или не выполняется вообще.

Плохой сценарий:

1. Заказ создался.
2. Первая позиция заказа сохранилась.
3. Остаток первого товара списался.
4. На второй позиции произошла ошибка.
5. В базе остался неполный заказ.

Так быть не должно.

Тест должен проверять не только ошибку, но и состояние после ошибки:

```python
def test_order_is_not_saved_partially_when_stock_is_not_enough(
    order_service,
    order_repository,
    stock_repository,
    cart,
):
    cart.add_item(product_id=1, size="M", quantity=10)

    with pytest.raises(ValueError):
        order_service.create_order(customer_id=1, cart=cart)

    assert order_repository.get_by_customer_id(1) == []
    assert stock_repository.get_quantity(product_id=1, size="M") == 5
```

Имена методов условные. В своей реализации используйте реальные имена.

## Как понять, что тест хороший

Хороший тест:

- имеет понятное название;
- проверяет конкретное поведение;
- быстро запускается;
- не зависит от порядка запуска других тестов;
- сам готовит нужные данные;
- сам проверяет результат;
- не требует ручного ввода;
- не ходит в рабочую базу;
- падает по понятной причине.

Плохие признаки:

- тест проверяет только "код не упал";
- в тесте нет `assert`;
- тест требует настоящего пользователя у клавиатуры;
- тест зависит от данных, которые кто-то вручную создал в базе;
- тест проходит только при запуске после другого теста;
- тест проверяет слишком много всего сразу.

## Покрытие тестами

Покрытие - это показатель, какая часть кода была выполнена тестами.

Для покрытия часто используют пакет `pytest-cov`:

```bash
python -m pip install pytest-cov
python -m pytest --cov=17_clothing_store_project
```

Высокий процент покрытия не гарантирует качество тестов.

Можно выполнить строку кода, но ничего важного не проверить.

Лучше думать не только о процентах, а о сценариях:

- что пользователь делает чаще всего;
- где можно потерять деньги или данные;
- где есть сложные условия;
- где уже были ошибки;
- какие операции меняют базу.

## Минимальный набор тестов для проекта магазина

Для раздела 17 минимально стоит покрыть:

1. Модели: создание корректных объектов и ошибки валидации.
2. Каталог: поиск, фильтрация, сортировка, скрытие неактивных товаров.
3. Корзину: добавление, изменение количества, удаление, сумма.
4. Скидки: активный, неактивный, неизвестный промокод, минимальная сумма.
5. Репозитории: `add`, `get_by_id`, `get_all`, `update`, `delete`.
6. Заказы: успешное оформление, нехватка товара, пустая корзина, откат при ошибке.
7. Финальный сценарий: покупатель находит товар, кладет в корзину, применяет промокод и оформляет заказ.

## Порядок работы

Удобный порядок такой:

1. Создать папку `tests/`.
2. Написать тесты на доменные модели.
3. Запустить pytest и добиться зеленого результата.
4. Написать тесты на корзину.
5. Написать тесты на сервисы через fake-репозитории.
6. Подготовить тестовую PostgreSQL-базу.
7. Написать интеграционные тесты репозиториев.
8. Написать интеграционные тесты оформления заказа.
9. Добавить тест полного пользовательского сценария.

Не обязательно писать все тесты за один раз. Хорошая практика - двигаться маленькими шагами: один сценарий, запуск, исправление, следующий сценарий.

## Главное

Тесты - это не отдельная бюрократия рядом с проектом.

Это способ описать поведение программы на языке кода:

- что должно работать;
- что должно быть запрещено;
- что считается ошибкой;
- какие данные должны сохраниться;
- какие данные не должны измениться при сбое.

Для проекта магазина особенно важно тестировать не только happy path, но и ошибочные сценарии. Магазин, который работает только при идеальном вводе, еще не готов. Готовая программа умеет корректно отказывать, не портить данные и оставаться предсказуемой.
