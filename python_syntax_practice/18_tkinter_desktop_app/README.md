# Большое задание. Desktop-интерфейс для онлайн-магазина одежды

## Идея проекта

Нужно постепенно разработать графический desktop-интерфейс на `tkinter` для проекта онлайн-магазина одежды из задания 17.

В задании 17 была создана backend-часть: модели, репозитории, сервисы, корзина, заказы и консольное приложение. В задании 18 нужно научиться подключать к этой логике другой интерфейс, не переписывая бизнес-логику под кнопки и поля ввода.

`tkinter` входит в стандартную библиотеку Python, поэтому отдельный веб-сервер, браузер, HTML, CSS и JavaScript здесь не нужны.

## Главная цель

Показать, что интерфейс - это отдельный слой приложения.

Desktop-приложение должно:

- показывать каталог товаров;
- позволять искать и фильтровать товары;
- добавлять товары нужного размера в корзину;
- менять количество товаров в корзине;
- показывать итоговую сумму;
- оформлять заказ через форму;
- показывать ошибки через окна сообщений;
- использовать код backend-слоя из задания 17.

## Архитектура проекта

Проект строится по этапам:

- `01_tkinter_basics/tasks.py` - первое окно, виджеты и главный цикл;
- `02_layout_widgets/tasks.py` - компоновка интерфейса и базовые `ttk`-виджеты;
- `03_events_state/tasks.py` - события, обработчики и переменные интерфейса;
- `04_desktop_architecture/tasks.py` - отделение GUI от бизнес-логики;
- `05_catalog_screen/tasks.py` - экран каталога товаров;
- `06_cart_screen/tasks.py` - экран корзины;
- `07_checkout_screen/tasks.py` - оформление заказа и формы;
- `08_final_desktop_app/tasks.py` - итоговая сборка desktop-приложения.

Это не веб-приложение и не MVC в строгом виде. Ближе всего проект устроен как `Domain + Services + Desktop UI`.

## Общие требования

1. Каждый этап выполняется в отдельной папке.
2. В каждом этапе есть `theory.md` и `tasks.py`.
3. Интерфейсный код не должен содержать SQL-запросы.
4. Интерфейсный код не должен напрямую менять внутренние списки товаров, заказов или остатков, если для этого уже есть сервисы.
5. Бизнес-логика не должна импортировать `tkinter`.
6. Обработчики кнопок должны быть короткими: получить данные из виджетов, вызвать сервис, обновить экран.
7. Ошибки пользователя нужно показывать через `messagebox`, а не завершать программу.
8. Для таблиц и списков используйте подходящие виджеты: `Listbox`, `Treeview`, `Combobox`.
9. Окно должно быть удобно использовать без консоли.
10. Финальное приложение должно запускаться отдельным Python-файлом.

Важно: в теории примеры приводятся на другой предметной области. Это сделано специально, чтобы студент понял принцип и применил его к магазину самостоятельно.

Так как папки курса начинаются с цифр, обычная запись вида `from 17_clothing_store_project...` может быть неудобной. Для учебного проекта можно использовать `importlib.import_module()` или аккуратно вынести общий код в импортируемые модули.

## Этапы проекта

### 01. Основы tkinter

Создать первое окно, добавить заголовок, размеры, несколько простых виджетов и запустить главный цикл приложения.

### 02. Компоновка и виджеты

Научиться собирать экран из `Frame`, `Label`, `Button`, `Entry`, `Combobox`, `Treeview` и размещать элементы через `pack()` или `grid()`.

### 03. События и состояние

Добавить обработчики кнопок, переменные `StringVar` и `IntVar`, чтение данных из полей и обновление интерфейса после действий пользователя.

### 04. Архитектура desktop-приложения

Разделить код на приложение, экраны, виджеты и слой вызова сервисов. Подготовить основу, которая сможет использовать backend из задания 17.

### 05. Экран каталога

Показать товары в таблице, добавить поиск, фильтры, просмотр выбранного товара и выбор размера.

### 06. Экран корзины

Реализовать добавление товара в корзину, изменение количества, удаление позиции, очистку корзины и пересчет суммы.

### 07. Экран оформления заказа

Добавить форму покупателя, адрес, промокод, проверку данных, создание заказа и понятные сообщения об ошибках.

### 08. Итоговое приложение

Собрать единое desktop-приложение с навигацией между каталогом, корзиной и оформлением заказа.

## Минимальный итоговый сценарий

Финальная программа должна позволять выполнить такой сценарий:

1. Открыть desktop-приложение.
2. Увидеть каталог активных товаров.
3. Найти товар по названию или категории.
4. Выбрать товар и размер.
5. Добавить товар в корзину.
6. Изменить количество или удалить позицию.
7. Посмотреть итоговую сумму.
8. Заполнить данные покупателя.
9. Применить промокод, если он есть.
10. Оформить заказ и увидеть подтверждение.

## Критерии готовности

Проект считается выполненным, если:

- каждый этап запускается без синтаксических ошибок;
- окно приложения открывается и закрывается корректно;
- интерфейс не содержит SQL-запросов;
- бизнес-логика не зависит от `tkinter`;
- каталог и корзина обновляются после действий пользователя;
- ошибки ввода не ломают приложение;
- оформление заказа работает через сервисы из backend-слоя;
- финальное приложение демонстрирует полный путь от каталога до заказа.
